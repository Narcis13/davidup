<design_notes>
## Schema Types vs Runtime Types

IMPORTANT: There are intentionally TWO keyframe representations:

1. **Schema types (Plan 03-02)**: KeyframeSchema uses `time` in SECONDS
   - User-friendly, fps-independent
   - Matches video editor conventions
   - Used in JSON input validation

2. **Runtime types (this plan)**: Keyframe interface uses `frame` in FRAME NUMBERS
   - Internal computation format
   - Precise, avoids floating-point drift
   - Used by animation engine

The AnimatedFrameGenerator (Plan 03-07) handles conversion from schema to runtime:
`frame = Math.round(time * fps)`

This is per RESEARCH.md recommendation: "Seconds in schema, convert to frames internally using video fps."
</design_notes>

  <behavior>
    interpolate(frame, inputRange, outputRange, options):
      - frame=0, input=[0,30], output=[0,100] -> 0
      - frame=15, input=[0,30], output=[0,100] -> 50
      - frame=30, input=[0,30], output=[0,100] -> 100
      - frame=-5, input=[0,30], output=[0,100], extrapolateLeft='clamp' -> 0
      - frame=35, input=[0,30], output=[0,100], extrapolateRight='clamp' -> 100
      - frame=-5, input=[0,30], output=[0,100], extrapolateLeft='extend' -> -16.67 (approx)
      - With easing=easeIn, frame=15 -> less than 50 (slow start)

    getAnimatedValue(frame, keyframes, defaultEasing):
      - Single keyframe: returns that value regardless of frame
      - Two keyframes [0->0, 30->100]: frame=15 -> 50 (linear)
      - Before first keyframe: returns first keyframe value
      - After last keyframe: returns last keyframe value
      - With keyframe easing: applies easing to that segment
      - Out-of-order keyframes: sorts internally, returns correct value

    getAnimatedElement(element, frame, fps):
      - Returns element with animated properties resolved for given frame
      - Handles startTime/endTime visibility (null if outside range)
      - Merges animated values with static element properties
  </behavior>
